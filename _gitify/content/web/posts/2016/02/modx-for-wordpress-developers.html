Plugins execute during certain system events (such as clearing the cache or saving a document).</p> <h3>Tag syntax</h3> <p>Here’s a quick reference of the MODX tag syntax:</p> <p><strong>Resource Fields - [[*field]]</strong><br> These are fields that are pulled from the current document, such as [[*pagetitle]] and [[*content]].</p> <p><strong>Template Variables - [[*name-of-tv:tag]]</strong><br> TVs are custom fields, again pulled from the current document.</p> <p><strong>Chunks - [[$name-of-chunk:tag]]</strong><br> Reusable HTML markup - think headers, footers, sidebars, or other common components.</p> <p><strong>Snippets - [[snippet:tag]]</strong><br> Dynamic bits of PHP.</p> <p><strong>Placeholders - [[+placeholder:tag]]</strong><br> Placeholders are typically used with snippets such as getResources to get another document’s resource field, however placeholders can be set to any value. Check out Mark’s article for a more detailed explanation of the difference between placeholders and resource fields.</p> <p><strong>Links - [[~1]]</strong><br> Generates a link to the resource. Replace 1 with the ID of the resource you want to link to.</p> <p><strong>System Settings - [[++system_setting]]</strong><br> These are settings that are configurable in the Manager.</p> <p><strong>Comments - [[- My comment. ]]</strong><br> Since you can’t use PHP in templates (with a Snippet), you’ll want to use this syntax if you want to remove your comments from the output.</p> <h3>Caching</h3> <p>MODX has excellent caching built in and caches everything by default. You can tell MODX not to cache a Snippet, Setting tag, Placeholder, or Chunk by adding an exclamation mark.</p><p>For example, the following would be cached:</p> <pre><code class="language-markup">[[$name-of-chunk]] [[helloWorld]]</code></pre> <p>And the following would <em>not</em> be cached:</p> <pre><code class="language-markup">[[!$name-of-chunk]] [[!helloWorld]]</code></pre> <p>In general, you want to cache as much as you can to improve the speed of your site. However, if what you’re calling changes on every request, you don’t want to cache it.</p><p>If you’re encountering issues where MODX isn’t updating the frontend as you’d expect, the first place to check is the cache. To clear the cache, in the Manager click ‘Manage’ and then ‘Clear Cache’.</p> <h2>WordPress anatomy</h2> <p>Now we’ve been through the building blocks of MODX, we can start to compare WordPress to MODX. Let’s jump in.</p> <h3>Themes</h3> <p>Themes don’t really exist in MODX. Well, they do sorta (you can <a href="http://themeforest.net/category/cms-themes/modx-themes" style="background-color: initial;">buy</a><a href="http://mdxthemes.com/" style="background-color: initial;">various</a><a href="http://mpthemes.com/" style="background-color: initial;">themes</a>). Out of the box, MODX has no way to import/export ‘themes’. There are various ways around this, such as creating a Transporter Package to bundle templates/TVs/chunks/snippets/etc.</p> <h3>Templates</h3> <p>Templates in MODX work much in the same way as they do in WordPress. The main difference is that WordPress dictates how you should organise and name your files. In MODX, you can organise and name things however you like.</p><p>My typically set up my folder structure to look something like this:</p><pre>assets/<br> templates/ (for static template files)<br> chunks/ (for static chunks)<br> snippets/ (for, you guessed it, more static files)<br> css/ (for compiled CSS from SASS files)<br> js/ (for compiled JavaScript files)</pre><p>You can of course organise this however you like.</p> <h2>Custom Post Types</h2> <p>CPT’s are effectively ‘buckets’ to store your content in. Let’s explore a simple way to achieve this in MODX.</p><p>Here’s an example of a Custom Post Type in WordPress:</p> <figure class="content-image extended-width "> <img data-src="/assets/uploads/images/2016/02/wp-cpt.png" class="lazyload" alt="" > </figure> <p>To create this in WordPress, you’d first create the CPT in functions.php. You’d then create an archive-books.php file to list the books and a single-books.php file to display individual books. The code might look something like this:</p><p><em>archive-books.php</em></p> <pre><code class="language-php"><h1>Books</h1> <ol> <?php while ( have_posts() ) : the_post(); ?> <li> <h2><a href="<?php the_permalink(); ?>"><?php the_title(); ?></a></h2> </li> <?php endwhile; ?> </ol></code></pre> <p><em>single-books.php</em></p> <pre><code class="language-php"><?php while ( have_posts() ) : the_post(); ?> <article> <h1><?php the_title(); ?></h1> <?php the_content(); ?> </article> <?php endwhile; ?></code></pre> <p>In MODX, one way to accomplish the same result is by using child resources and the getResources Snippet. First, install getResources by going to Extras > Installer in the MODX manager.</p><p>Next, set up the resource tree like this:</p><p>{screenshot}</p><p>The Book resource would have it’s own template which would include a getResources call. The template would look something like this:</p> <pre><code class="language-markup"><h1>[[*pagetitle]]</h1> <ol> [[getResources? &parents=`[[*id]]` &tpl=`book-list-item` &includeContent=`1` &limit=`10` ]] </ol></code></pre> <p>In the example above, we’re calling the getResources Snippet and passing a couple of properties. First, we’re specifying the parent as the current page so that it loops through the child resources. Next, we’re telling the Snippet to use the `book-list-item` chunk. `&includeContent` tells the Snippet to return the content of each resource and finally we’re specifying a limit of 10 resources. There’s <a href="https://rtfm.modx.com/extras/revo/getresources" style="background-color: initial;">more properties</a> you can pass to getResources.</p><p>The `book-list-item` chunk would then look like this:</p> <pre><code class="language-markup"><li> <h2><a href="[[~[[+id]]]]">[[+pagetitle]]</a></h2> </li></code></pre> <p><em>Note: in the chunk above we’re using placeholders, not resource fields. This is because we want to pull the fields from the child page, not the current page.</em></p><p>Each individual resource would have a template (equivalent to single-books.php) that would look something like this:</p> <pre><code class="language-markup"><article> <h1>[[*pagetitle]]</h1> [[*content]] </article></code></pre> <h3>Custom Fields</h3> <p>In our book example above, we’ll probably want to add some custom fields such as ‘author’ and ‘ISBN’.</p><p>In WordPress, you could create a custom field by using the Advanced Custom Fields plugin.</p><p>`single-books.php` would then look like this:</p> <pre><code class="language-php"><?php while ( have_posts() ) : the_post(); ?> <article> <h1><?php the_title(); ?></h1> <p>Author: <?php the_field(‘author’); ?></p> <p>ISBN: <?php the_field(‘isbn’); ?><?p> <?php the_content(); ?> </article> <?php endwhile; ?></code></pre> <p>In MODX, you’d use Template Variables, which are assigned to specific templates. In this case, we’d assign our Template Variables to the `Book Single` Template.</p><p>First, in order to pass through template variables, we’ll need to modify our getResources call by setting includeTVs and processTVs to 1:</p> <pre><code class="language-markup">[[getResources? &parents=`[[*id]]` &tpl=`book-list-item` &includeContent=`1` &includeTVs=`1` &processTVs=`1` &limit=`10` ]]</code></pre> <p>Our code would then look like this:</p> <pre><code class="language-markup"><article> <h1>[[+pagetitle]]</h1> <p>Author: [[+author]]</p> <p>ISBN: [[+isbn]]<?p> [[+content]] </article></code></pre> <p>Easy, huh?</p> <h3>Taxonomies</h3> <p>MODX doesn’t really have a concept of taxonomies. You could use Template Variables to accomplish this or use a <a href="https://modx.com/extras/package/taxonomies" style="background-color: initial;">plugin</a>.</p> <h3>The Loop</h3> <p>The Loop in WordPress returns the appropriate content for the given page you’re on. You might loop through blog posts like this:</p> <pre><code class="language-php"><?php while ( have_posts() ) : the_post(); ?> <h1><a href="<?php the_permalink(); ?>"><?php the_title(); ?></a></h1> <?php the_content(); ?> <?php endwhile; ?></code></pre> <p>In MODX, we could again use the getResources Snippet to achieve the same result:</p> <pre><code class="language-markup">[[getResources? &parents=`[[*id]]` &tpl=`article-list-item` &includeContent=`1` ]] </code></pre> <p>The getResource Snippet above calls the chunk `article-list-item`, which would look like this:</p> <pre><code class="language-markup"><h1><a href="[[~[[+id]]]]">[[+pagetitle]]</a></h1> [[+content]]</code></pre> <p><em>Note: in the chunk above we’re using placeholders, not resource fields. This is because we want to pull the fields from the child page, not the current page.</em></p> <h3>Menus</h3> <p>To create a dynamic menu in WordPress, you’d register the menu in `functions.php` and then use the following in your template:</p> <pre><code class="language-php"><?php wp_nav_menu( array( 'menu' => 'Main Primary Menu’ ) ); ?></code></pre> <p>In MODX, a common way to do this is to use the <a href="https://rtfm.modx.com/extras/revo/wayfinder" style="background-color: initial;">Wayfinder</a> extra. Once installed, you can call the Snippet like this:</p> <pre><code class="language-markup">[[Wayfinder? &startId=`0` &level=`1` &outerTpl=`menu.outer` &rowTpl=`menu.row`]]</code></pre> <p>In a similar way to getResources, we’re passing a couple of properties such as startId (the top level parent) and level (how many nested resources to go down). We’re also using two chunks to build the menu. There’s a bunch more <a href="https://rtfm.modx.com/extras/revo/wayfinder" style="background-color: initial;">properties</a> available to customise the menu.</p><p>Here’s what the `menu.outer` could look like:</p> <pre><code class="language-markup"><nav role="navigation" class="primary-nav"> [[+wf.wrapper]] </nav></code></pre> <p>And `menu.row`:</p> <pre><code class="language-markup"><a href="[[+wf.link]]" title="[[+wf.title]]" [[+wf.classes]]>[[+wf.linktext]]</a></code></pre> <p>Hide from menu in resources.</p> <h3>Multi-site</h3> <p>My experience of building a <a href="http://codex.wordpress.org/Create_A_Network" style="background-color: initial;">WordPress multi-site</a>, was.. well, let’s just call it interesting.</p><p>Although this article isn’t a “MODX is better than WordPress” article, this is a case where MODX is clearly superior.</p><p>MODX has “<a href="https://rtfm.modx.com/revolution/2.x/administering-your-site/contexts">Contexts</a>” which allows you to build multiple sites that share elements. This means that you can host multiple sites, subdomains, landing pages, etc. on the same install and manage via the same Manager. Using MODX user permissions, you can create access groups to only access certain contexts if required.</p> <h3>Multi-lingual</h3> <p>There’s multiple ways to handle multi-lingual sites in MODX. One way is to use “Contexts” as mentioned above to create unique resources for each language, while sharing elements.</p> <h2>Recommend Extras to get started</h2> <p><a href="https://rtfm.modx.com/extras/revo/wayfinder" style="background-color: initial;">Wayfinder</a> -
