You know MODX gives you creative freedom enabling you to control each character of the markup delivered to your audience. MODX is considered by many to be the best suited system to create responsive and accessible web experiences. But the web shouldn't just be something anyone can browse — it should be something anyone can author as[[$nbsp]]well.

On March 21st the Wordpress team made an incredible announcement that will make the web a more inclusive place to[[$nbsp]]author.

All new or updated code released into WordPress core and bundled themes must conform with the WCAG 2.0 guidelines at level AA.

The Wordpress project has taken accessibility seriously for years. Their Dashboard is mobile friendly, works without JavaScript enabled, and uses accessible icons. So what is so significant about their recent announcement? By requiring WCAG 2.0 at level AA Guidelines for any updates Wordpress has drawn a line in the sand. They have made a clear and concise commitment to the World Wide Web. They aren't just striving for accessibility — they are requiring it. This means that new additions to the core software must meet the WCAG guidelines or they will not be accepted into the core. No[[$nbsp]]exceptions.

One can safely assume Wordpress is willing to make this commitment with the realization that accessible software is better for everyone. They may loose out on or prolong the adoption of shiny new features but they must rest easy knowing they ship inclusive software. Publishing to the web is the ability to share your expressions, passions, and thoughts with the world. The ability to be heard, seen, understood, and followed is a right — not a privilege. Unfortunately, the web isn't always built this way. We can do better. But[[$nbsp]]how?

With a single sentence Wordpress makes a clear and concise commitment that leaves no room for speculation as to how accessibility is treated in their core software — with an entire campaign MODX does[[$nbsp]]not. Wordpress officially commits to the WCAG Guidelines Level AA for their core and anything bundled with the product. With the exception of addressing color contrast ratios in a separate theme, MODX does not commit to WCAG Guidelines of any level. MODX does not commit – beyond hope – to improving accessibility in its own[[$nbsp]]core. Wordpress commits to directly improving accessibility through future updates of their core products. Wordpress supports people. MODX supports particular[[$nbsp]]users.

Make Clear & Concise Commitments

Hope is not enough. There is no commitment that pull requests which make the MODX core more accessible will wind up in the a11y theme. Likewise, there is no official commitment that improvements made in the a11y theme will work their way back into the MODX core. We have issues with labels and hopes. Issues labeled Port to Revo are closed regardless of if they have actually been ported to Revo. Accessibility issues are closed without being address in the core. This causes confusion and leaves contributors unsure if they should submit pull requests to the core, the a11y theme, or neither. As an open source project let us come together and commit to globally adopted standards without[[$nbsp]]confusion.

We need to take a look at how the rest of the industry solves these[[$nbsp]]problems and integrate them directly into the core of future[[$nbsp]]releases. Let us learn from Wordpress by making clear and concise[[$nbsp]]commitments. Let us focus on our core and people rather than bolt–on themes and[[$nbsp]]users.

Support People Not Users

Your abilities are not just relative to your person they are also relative to the devices you access experiences through. Together your person and a connected device create a user. There is a misconception that there are normal people and then there are people with disabilities whom accessibility applies to. Let us address this. Accessible software is better for everyone, not just a targeted[[$nbsp]]group.

The final result will be a new Manager Theme you can install in MODX 2.3 or later. A simple change of your Manager Theme setting will enable anyone to use MODX to create and manage sites in an accessible CMS.  — a11y.modx.com

Enable anyone to use MODX? No it won't. What if a person is using a touch device, mobile device, or doesn't receive JavaScript even though it is enabled? What if they can't navigate to the Package Manager to install the opt–in theme in the first[[$nbsp]]place? We cannot skip over HTML — and our own core — in our development process and support anyone and[[$nbsp]]everyone. The web is for everyone and it was designed to be built atop HTML for that very[[$nbsp]]reason. Let us go with the grain of the web not against[[$nbsp]]it.

Web accessibility means that people can use the web — not particular users. Let us learn from Wordpress and put the notion that assistive technology is only applicable to a particular group to rest. Let us work towards a web all people can author with[[$nbsp]]MODX.

[MODX] Next Please

The MODX 2.x Dashboard was not designed to be inclusive. You'll hear no argument from me that it should have been either. Let me explain. When I first discovered MODX 2.0.8 I assumed it was software lead by a team of dozens of developers who focused full–time on the product. That is not and never has been the case. To this day, MODX 2.x leads the industry in the areas it has focused on. It is nothing sort of spectacular. Jason Coward and Shaun McCormick, along with a small group of contributors, managed to create something truly brilliant with a fraction of the resources necessary to do[[$nbsp]]so.

By any reasonable measure MODX Revolution should be considered a complete success. But I'm sick of being reasonable. Aren't you? It is time we face another defining challenge. Let us enable anyone to author the web with MODX. Let us reimagine the Manager interface from the ground up. Let us commit clearly and concisely that in a future major version (MODX Next) the core must comply with the WCAG Guidelines. No exceptions. Let us join Wordpress in the commitment to make authoring the web inclusive for[[$nbsp]]everyone. Let us do so directly in future updates to our software. Let us learn from[[$nbsp]]Wordpress.
